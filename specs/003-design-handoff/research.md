# Phase 0 Research: Design Handoff Adoption

**Feature**: 003-design-handoff
**Date**: 2026-04-20

Every open technical decision the plan touched is resolved below. Each entry records what was picked, why, and what was rejected — so future readers can replay the decision without re-running the investigation.

---

## 1. Token file format: keep `colors_and_type.css` as `.css` vs. transform to `.ts`

**Decision**: Keep tokens as plain CSS (`apps/frontend/src/styles/tokens.css`), copied verbatim from the handoff. TypeScript consumers reference tokens through a small `tokens.d.ts` that types the valid CSS variable names as a string-literal union.

**Rationale**:

- The handoff's `colors_and_type.css` is already a complete, valid stylesheet including `@font-face` (post-rewiring), `:root`, keyframes, and base selectors (`.p`, `.h1`, `.eyebrow`, etc.). Converting it to TypeScript would require re-serializing it back to CSS at build time or losing the non-variable declarations (body resets, semantic type classes).
- The constitution's Principle III explicitly allows `tokens.css` as a valid token source.
- CSS variables are consumed natively by every style surface: plain CSS, CSS Modules, inline `style={{}}`, Tailwind theme values via `var(--x)`.
- Diffs against future handoff deliveries are line-for-line meaningful in CSS; a TS transform would obscure them.

**Alternatives considered**:

- `design-tokens.ts` with every value as a TypeScript constant — rejected. Forces double-declaration (`--accent-yellow` in CSS and `AccentYellow` in TS) and a bespoke build step.
- `tokens.css` + a TS export of the variable names generated by a codemod — deferred. A single `tokens.d.ts` (hand-maintained) is enough surface for Phase 1; codegen can land later if the variable list grows past ~100.

## 2. Tailwind integration strategy

**Decision**: Install Tailwind v3.4 with Preflight disabled (`corePlugins.preflight: false`). The `tailwind.config.ts` theme extension wraps CSS variables (e.g., `colors: { ink: { 900: 'rgb(var(--ink-900-rgb) / <alpha-value>)' } }` or the simpler `var(--ink-900)` form for cases that don't need alpha modifiers). Content scan covers `apps/frontend/src/**/*.{ts,tsx,css}` excluding `showcase/` when building for production.

**Rationale**:

- The constitution's Frontend Stack list names Tailwind explicitly.
- Preflight must be disabled because `tokens.css` already owns `body`/`.p`/`::-webkit-scrollbar-*`/reset declarations; letting Tailwind's Preflight layer on top would break the handoff's type defaults and scrollbar treatment.
- Wrapping CSS variables (rather than redeclaring values) keeps `tokens.css` the single source of truth per FR-005 — changing `--accent-yellow` in one place still propagates through Tailwind utilities.
- Tailwind v3 is stable and matches the ecosystem that Yalgamers' tailwind.config.ts already uses. v4 (CSS-first) is promising but newer; defer to a future upgrade.

**Alternatives considered**:

- No Tailwind, CSS Modules only — rejected. Contradicts the constitution's pinned stack.
- Tailwind v4 CSS-first — rejected for Phase 1. Adds migration risk for a greenfield phase; we can revisit when the ecosystem stabilizes.
- Two-layer tokens (TS for Tailwind theme + CSS for variables, kept in sync) — rejected. Violates FR-005's single-source rule.

**Alpha-modifier note**: to support Tailwind's `<alpha-value>` syntax (`bg-ink-900/50`), the theme maps take the form:

```ts
colors: {
  ink: {
    900: 'rgb(from var(--ink-900) r g b / <alpha-value>)',
  },
}
```

The `rgb(from ...)` function is broadly supported in evergreen browsers (2024+). For tokens we don't need alpha modifiers on, the simpler `var(--x)` reference works.

## 3. Routing: `react-router-dom` now vs. a pathname switch

**Decision**: No router in Phase 1. `main.tsx` checks `window.location.pathname === '/_showcase' && import.meta.env.DEV` and renders either the production `<App />` placeholder (Phase 0) or `<Showcase />`. Phase 6 replaces this with `react-router-dom` v6.

**Rationale**:

- The constitution names `react-router-dom` v6 but as the overall stack pick, not a phase-specific constraint.
- Phase 1 has exactly one non-production "route". A full router is overkill and pulls Phase 6 forward without the accompanying guards, persistence, and auth flow that make routing worthwhile.
- Vite's DCE (via Rollup) reliably drops the `import('./showcase/Showcase')` dynamic import when `import.meta.env.DEV` is `false` at build time, keeping SC-010 honest.

**Alternatives considered**:

- Install `react-router-dom@6` now — rejected. Scope creep; Phase 1 doesn't need its features.
- Separate dev entry (`dev-main.tsx` + `index.dev.html`) — rejected. Two HTML entry points complicate `pnpm dev` and drift from Phase 0's layout.

**Implementation sketch**:

```tsx
// main.tsx
const rootEl = document.getElementById('root');
if (!rootEl) throw new Error('#root not found');

if (import.meta.env.DEV && window.location.pathname === '/_showcase') {
  import('./showcase/Showcase').then(({ Showcase }) => {
    createRoot(rootEl).render(
      <StrictMode>
        <Showcase />
      </StrictMode>,
    );
  });
} else {
  createRoot(rootEl).render(
    <StrictMode>
      <App />
    </StrictMode>,
  );
}
```

## 4. Font self-hosting: source, subsetting, and `font-display`

**Decision**: Download `.woff2` files for Mulish (weights 400/500/600/700/800), Inter (weights 400/500/600/700), and Bruno Ace SC (weight 400) from Google Fonts Helper (`gwfh.mranftl.com`) in the Latin subset only. Commit them under `apps/frontend/public/fonts/` and declare them with `@font-face` in `tokens.css`, all with `font-display: swap`.

**Rationale**:

- Mulish, Inter, and Bruno Ace SC are all Open Font License fonts — legally redistributable in the repo.
- Latin subset is sufficient for the design (no Cyrillic/Arabic/CJK in scope).
- `font-display: swap` is the right default for a gaming UI: the layout is dominated by the display and UI fonts; showing a system fallback for ≤100ms while `.woff2` decodes is better than a blank page (FOIT).
- Google Fonts Helper generates stable, version-pinned URLs and the exact `@font-face` CSS to copy.

**Alternatives considered**:

- `fontsource-*` npm packages (e.g., `@fontsource/mulish`) — viable but adds dependencies and indirection. Direct `.woff2` in `public/` is simpler and deterministic.
- Variable fonts (Mulish Variable) — tempting (single file, every weight), but Bruno Ace SC is not available as a variable font, so we'd have mixed strategies. Use static weights across the board.
- `font-display: block` for the display font — rejected. Blocks render for up to 3 seconds; the flash of fallback is the lesser evil.

**File naming convention**: `<family>-v<version>-<subset>-<weight>.woff2` — matches Google Fonts Helper's output so a bundle refresh is a drop-in replace.

## 5. Reduced-motion strategy

**Decision**: Wrap every `animation:` or `transition:` property that references a named keyframe in a `@media (prefers-reduced-motion: no-preference)` block. The default (outside the media query) is the no-animation state. A showcase-only override: when `<html data-reduced-motion="true">` is set by the `ReducedMotionToggle`, the animations collapse even if the OS preference is `no-preference`.

**Rationale**:

- The `@media (prefers-reduced-motion: no-preference)` inversion is the recommended pattern: writing the motion-ful rules inside the query means the "static" experience is the default and animations are the progressive enhancement. This guarantees that if a developer forgets to add the guard to a new animation, reduced-motion users still get motion (a bug to fix on review) rather than a motionless experience for `no-preference` users (a regression).
- Wait — restated correctly: the right pattern is to put motion rules inside `@media (prefers-reduced-motion: no-preference)`. Default = reduced. That way users with the preference (or no preference signal either way from older browsers) get the safer path.
- The showcase's synthetic toggle (`data-reduced-motion="true"`) is not about OS preference; it's a reviewer's manual lever to verify the reduced-motion path without toggling OS settings. The CSS respects both signals with `@media (prefers-reduced-motion: no-preference) { html:not([data-reduced-motion="true"]) .cell-claimed { animation: claimPulse ... } }`.

**Alternatives considered**:

- Put motion rules unguarded and add `@media (prefers-reduced-motion: reduce) { * { animation: none !important; } }` — rejected. The nuclear reset is heavy-handed and silently kills third-party animations (e.g., Lucide icon loading spins) that might be intentional. Per-animation guards give precision.
- JavaScript-driven motion gating via `matchMedia` — rejected. Adds a runtime code path for what CSS handles declaratively.

**Information preservation**: FR-013a requires that the underlying information is preserved when motion collapses. Concretely:

- `claimPulse`: without motion, the cell still flips color (`background-color` change is immediate). Only the scale/brightness pop is suppressed.
- `timerPulse`: without motion, the box-shadow glow is static at its peak intensity. Color transitions remain.
- `gridDrift`: without motion, the backdrop is static.
- `fadeUp`: without motion, elements are already at their final position/opacity (no opacity tween).

## 6. Icon strategy: handoff custom icons + Lucide

**Decision**: Per the Q4 clarification, ship two sources through one export surface.

- **Brand icons** (hand-authored SVG React components under `components/common/icons/brand/`): Crown-host indicator, Zonite logo, Yalgamers logo, and any other icon the handoff's `Icons.jsx` defines that is brand-specific (Z mark, fire-gradient swatches).
- **Generic icons** (Lucide, imported via `lucide-react`): every other UI glyph (copy, close, chevron, check, search, info, warning, arrow, settings, eye / eye-off, check-circle, x-circle).
- **Single export**: `components/common/icons/index.ts` re-exports both sets under the same `<IconFoo />` API naming. Callers import from `@/components/common/icons` only; neither source leaks.

**Rationale**:

- Lucide v0.460+ ships tree-shakable named exports. Importing `CheckIcon` only adds the one icon (and its parent `createLucideIcon` helper) to the bundle — small and predictable.
- Brand icons as hand-authored SVG components preserve the exact stroke and fill design ships wants; a library equivalent for "crown + laurel" (e.g., `IconCrown` in Lucide) does not match the handoff's custom brand mark.
- One barrel file means a reviewer can see every shipped icon in one `index.ts` — makes the FR-010b override-policy rule "reviewer blocks a PR that re-draws a generic glyph or swaps a brand icon" enforceable on a single file diff.

**Alternatives considered**:

- Heroicons — rejected. Style is outlined-only and heavier than Lucide; Lucide has denser weight options.
- `@radix-ui/react-icons` — rejected. Fewer glyphs and tied to Radix.
- Drop Lucide, ship only handoff icons — rejected. Phase 7 needs dozens of generic glyphs the handoff won't pre-draw, and re-drawing each is wasted work.

## 7. Stylelint and custom ESLint rule: scope and rule set

**Decision**: Enforce "no hex / no raw font-family" via two layers with narrow, opinionated scope.

**Stylelint** (`apps/frontend/stylelint.config.cjs`):

```js
module.exports = {
  extends: ['stylelint-config-standard'],
  plugins: ['stylelint-order'],
  rules: {
    'color-no-hex': [true, { severity: 'error' }],
    'font-family-name-quotes': 'always-where-recommended',
    'declaration-property-value-disallowed-list': {
      'font-family': ['/^(?!var\\(--font-).*/'], // every font-family MUST reference a --font-* token
    },
  },
  ignoreFiles: ['**/styles/tokens.css'], // the token file itself is the only place hex + font-family live
};
```

**Custom ESLint rule** (`eslint-rules/no-hex-in-jsx.js`): scans `JSXAttribute` nodes named `style` with an object expression value; for each property, regex-matches `#[0-9a-fA-F]{3,8}` in string literals and fails with "Hard-coded hex color in inline style; use a token from tokens.css." A second check matches `fontFamily` keys where the value is not exactly `'var(--font-ui)' | 'var(--font-display)' | 'var(--font-mono)'`.

**Rationale**:

- Stylelint is the right tool for CSS / CSS Modules files; its ecosystem is mature.
- ESLint's default rules don't cover inline-style regex checks; a local rule file (registered in `eslint.config.mjs` via the flat config's `plugins` field) is ~50 lines of AST-walk code and costs us no dependency.
- Both rules point engineers at `tokens.css` by name so the fix is obvious.
- Scoping the rules to `apps/frontend/src/**` keeps them from firing on config files and the handoff bundle under `docs/design/`.

**Alternatives considered**:

- `stylelint-no-restricted-syntax` for inline styles in TSX — rejected. Stylelint doesn't parse JSX; wrong tool.
- Skip the custom ESLint rule, rely on a CI grep — rejected. Grep misses regex edge cases (e.g., hex with alpha, e.g., `#ffee0080`) and doesn't surface failures in the IDE during authoring.
- Block all inline `style={{}}` entirely — rejected. Some primitives legitimately need dynamic style (e.g., Countdown's remaining-percentage fill). The rule targets values, not the use of inline styles.

## 8. Bundle SHA verification: hashing algorithm and CI integration

**Decision**: Ship `scripts/verify-handoff.mjs` as a small Node script (no external deps) that:

1. Walks `docs/design/zonite-game/` recursively.
2. Sorts the file list lexicographically for determinism across platforms.
3. For each file: reads its content, appends `path + '\0' + content` to a running SHA256 hash.
4. Compares the hex digest to the `expected_sha256` field in `docs/design/HANDOFF_VERSION.md`.
5. Exits non-zero (with a human-readable diff summary) if they do not match.

A CI workflow step runs `node scripts/verify-handoff.mjs` on every PR. A pre-commit hook can optionally run it (configurable in husky).

**`HANDOFF_VERSION.md` format**:

```markdown
# Zonite Design Handoff — Version Manifest

adopted_at: 2026-04-20
bundle_source: Claude Design (claude.ai/design) — export from session 53AOnvUnv452coICdMiQ_w
expected_sha256: <64-char hex>

## Refresh workflow

When the design team publishes a new bundle:

1. Replace `docs/design/zonite-game/` with the new tarball's contents.
2. Run `node scripts/verify-handoff.mjs --record` to update `expected_sha256` in this file.
3. Diff `colors_and_type.css` into `apps/frontend/src/styles/tokens.css` — preserve the self-hosted `@font-face` rewire.
4. Diff prototype components into primitives if design changed them.
5. Submit the refresh in a single PR that also updates this file.
```

**Rationale**:

- Hashing `path + '\0' + content` (null-separator) prevents a boundary-collision attack where renaming two files could produce the same concatenated byte stream as the original.
- Sorting lexicographically makes the hash identical on Linux and macOS dev machines.
- Using Node's built-in `crypto` module + `fs/promises` keeps `package.json` free of new deps.
- A `--record` flag lets the refresh author regenerate the hash without manually running `sha256sum`.

**Alternatives considered**:

- Tar the bundle tree and hash the tarball — rejected. `tar` output is not bit-stable across platforms (differing uid/gid, mtime, extended attrs).
- Git's tree SHA — rejected. Changes if any file's line-ending differs; too brittle for a cross-platform dev setup.
- `sha256sum --check` against a manifest of per-file hashes — viable but noisier to maintain. The single aggregate hash suffices for "did any file change" detection.

## 9. Axe-core integration for accessibility verification

**Decision**: Dynamic-import `@axe-core/react` inside `Showcase.tsx` only when `import.meta.env.DEV` is true. On mount, call `axe.run(rootEl, { runOnly: ['wcag2a', 'wcag2aa', 'wcag21aa'] })` and log failures to the console with a summary banner at the top of the showcase. The showcase also renders an "axe report" panel that lists violations inline, so reviewers see them at a glance.

**Rationale**:

- `@axe-core/react` is dev-dependency only; dynamic import means production builds never pull it in.
- Running axe against the showcase specifically (rather than the whole app) verifies every primitive in every state that matters, which is what SC-009 audits.
- WCAG 2.1 AA is the declared baseline (FR-013a); the axe config pins that exact rule set.

**Alternatives considered**:

- `@axe-core/cli` run from CI against a static-built showcase — too much infrastructure for Phase 1 (would need headless Chrome in CI). Defer to a later phase if we want automated a11y gates.
- `eslint-plugin-jsx-a11y` — already likely included via common React linting, but it's static analysis and can't check runtime color contrast. Keep it on; add axe for runtime.
- Pa11y — similar to axe but less React-integrated. Axe has the mature React wrapper.

## 10. Contrast auditing: which surfaces count as "documented"?

**Decision**: Every primitive that displays text or an interactive glyph declares its "documented default surface" (the background the primitive expects to be placed on) via a comment at the top of its `.module.css` file:

```css
/* Primitive: Button (primary variant)
 * Documented surface: --bg-page / --bg-card / --bg-elevated
 * Text color: --fg-primary on --accent-yellow (contrast 14.1:1 against yellow, 16.5:1 against ink-900 when ghost)
 */
```

The `primitives.contract.md` pins the full matrix: for every primitive × variant × state, the expected foreground/background token pair and the pre-computed contrast ratio. The axe pass runs against the showcase where each primitive is rendered on its documented surface; any other placement is the caller's responsibility.

**Rationale**:

- "WCAG AA on the documented surface" is precise and auditable; "WCAG AA everywhere" is unachievable without owning every consumer context.
- A pre-computed ratio per variant lets reviewers catch regressions in handoff updates (if design darkens `--ink-900`, the `--fg-primary` contrast might climb; if they lighten it, it might drop below 4.5:1 — the matrix surfaces the delta).

**Alternatives considered**:

- Skip pre-computed ratios, rely on axe at runtime — rejected. Runtime only catches what's currently rendered; the matrix is a design-time check that forces the conversation with design when a ratio fails.

## 11. Keyboard operability minimum per primitive

**Decision**: Specified in `primitives.contract.md` as the "Keyboard" column for each interactive primitive. Summary:

- `Button`, `SegButton`, `Chip` (interactive): Enter / Space activates; Tab reaches; visible focus via `--focus-ring`.
- `Input`, `Field`: native focus; Esc clears (only if the field has a clear button rendered).
- `OtpField`: Tab reaches the first slot; digit entry auto-advances; Backspace on an empty slot auto-reverses; arrow keys move between slots.
- `Slider`: Arrow Left/Right = ±step; Home / End = min / max; Page Up / Down = ±10×step.
- `Modal`: Focus-trap inside the modal while open; Esc closes (unless explicitly disabled); restore focus to the trigger on close.
- `Alert` (dismissible variant): dismiss button is focusable and activated by Enter/Space.
- `Avatar` (clickable variant): treated as `Button` when `onClick` is provided; otherwise non-interactive.

**Rationale**: This is the minimum to pass SC-009's manual keyboard walk-through. Anything richer (e.g., combobox keyboard patterns) lives in later phases' primitives, not these.

**Alternatives considered**:

- Adopt Radix UI primitives for the interactive ones — considered. Radix is excellent but adds a large dep + API surface that later phases would inherit without being ready to leverage. The custom primitives match the handoff's look exactly, and the keyboard patterns above are well-trodden enough to implement by hand.

## 12. Screenshot pipeline for the Spekit design-system Spek

**Decision**: Capture screenshots manually from the handoff prototype (`docs/design/zonite-game/project/Zonite App.html` rendered in a Chromium browser at 1280×800 viewport) at Phase 1 implementation time. Store them under `docs/design/screenshots/` as PNGs named `zonite-<screen>.png` (e.g., `zonite-onboarding.png`). The Spekit design-system Spek embeds or links each.

**Rationale**:

- The bundle README says "Don't render these files in a browser or take screenshots unless the user asks you to." — but the constitution's Spek requirement (FR-026) is an explicit ask: the Spekit reference needs visual previews.
- Manual capture once, at a fixed viewport, keeps the screenshots stable. Automated capture (Playwright headless) is overkill for a one-time artifact.

**Alternatives considered**:

- Skip screenshots, link to the prototype HTML directly — rejected. Spekit consumers need inline previews; opening the prototype requires a dev setup.
- Automate with Playwright — rejected. Phase 1 ships no test tooling; introducing headless browsers for one-time screenshots is disproportionate.

## 13. Override policy canonical document

**Decision**: Publish the override policy as `docs/design/OVERRIDE_POLICY.md` and mirror it into the Spekit Spek "Zonite Design System — token sources and override policy". The Markdown file IS the source; the Spek embeds or links it.

**Structure** (from FR-019):

```markdown
# Zonite Frontend Override Policy

## When to add a new token

- The value will be used in ≥2 places or by a brand-level primitive.
- You have design sign-off (or the value came from a handoff refresh).
- Add to `apps/frontend/src/styles/tokens.css` under the right category.
- Update `tokens.d.ts` and (if used) the Tailwind theme extension.

## When a local style exception is permissible

- Exactly-one-off and justified in an adjacent comment: `/* exception: … */`.
- The value is computed at render time (e.g., `style={{ width: `${n}%` }}` for a slider fill).
- Reviewer explicitly approves in the PR description.

## What reviewers block

- Raw hex / font-family / pixel-spacing in TSX or CSS outside `tokens.css`.
- Re-declaring a token value in Tailwind theme (vs. wrapping the CSS variable).
- Swapping a brand icon for a Lucide equivalent (or vice versa).
- A primitive that ships without declaring its documented surface.
- An animation added without a `prefers-reduced-motion: no-preference` guard.
```

**Rationale**: Putting the policy under `docs/design/` co-locates it with the bundle and the version manifest. The Markdown is the canonical text; the Spek is the discoverability surface.

**Alternatives considered**:

- Policy only in Spekit — rejected. Spekit is external; losing network access shouldn't lose the policy. On-repo copy is authoritative.
- Policy only in `README.md` — rejected. Policy is long enough to deserve its own file; README stays as a landing page.
